# Construction Bot

## 1. Что это

Construction Bot — телеграм-бот, который раз в день собирает изменения по стройкам пиццерий из
задач Битрикс24 (комментарии, статусы, чек-листы, файлы) и присылает короткую выжимку в топик
странового чата партнёра. Вместо того чтобы владелец сети или партнёр вручную открывали карточку
задачи в Битриксе, они читают одно сообщение в Telegram: что обсуждали за день, какие решения или
блокеры появились, какие файлы приложили.

Бот работает по подписке «топик ↔ карточка»: один топик может отслеживать несколько карточек
(строек), одна карточка может быть подписана в нескольких топиках. Если за сутки по карточке не
было ничего нового — дайджест по ней не приходит (чат не заваливается пустыми сообщениями); если
молчание тянется дольше недели — раз в неделю приходит короткое напоминание, что подписка жива.
Всё управление — через команды прямо в чате, без отдельной админки.

## 2. Поверхности

### Команды бота

| Команда | Что делает |
|---|---|
| `/add <task_id ИЛИ ссылка>` | Подписывает текущий топик на карточку Битрикса. Карточек миллион — поиск по названию не поддерживается: принимает голый ID (`/add 8017`) либо ссылку на карточку, скопированную из браузера (ID вытаскивается регэкспом из `.../tasks/task/view/<id>/`). Если карточка не найдена или нет доступа — сообщает об этом. Повторный `/add` на уже отслеживаемую карточку — no-op с уведомлением; на ранее снятую — реактивирует подписку. Без аргументов — открывает диалог (reply-промпт), а не подсказку «Использование». Подзадачи такой карточки в Битриксе бот подхватывает **сам**, на каждом прогоне дайджеста — отдельный `/add` на них не нужен. |
| `/remove <task_id>` | Снимает подписку топика с карточки. Если это ручная карточка — заодно снимает и её авто-подхваченные подзадачи. Если карточка не отслеживалась — сообщает об этом. Без аргументов — открывает клавиатуру со списком карточек (тот же флоу, что кнопка «🗑 Убрать» в `/menu`). |
| `/list` | Показывает список карточек, отслеживаемых в текущем топике. Авто-подхваченные подзадачи помечены «↳» перед названием (вместо обычного «•») — видно, что бот завёл карточку сам. |
| `/lang <код>` | Меняет язык дайджеста для чата (`ru`, `en`, ...). Влияет и на язык, на котором LLM пишет выжимку. Без аргументов — открывает клавиатуру выбора языка. |
| `/menu` | Панель с кнопками: добавить/убрать карточку, отчёт сейчас, время, язык — вся работа тыканьем, без адресных команд. Ввод номера карточки — reply на сообщение бота. |
| `/report` | Дайджест прямо сейчас, не дожидаясь дневного расписания (кнопка «📊 Отчёт сейчас» делает то же). Дневной дайджест не ломает. |
| `/time <ЧЧ:ММ> [таймзона]` | Задаёт локальное время дайджеста для чата. Таймзону нужно указать явно при первой настройке (пока чат на дефолтной UTC); дальше можно менять только время, сохраняя ранее заданную таймзону. Без аргументов — открывает reply-промпт (тот же флоу, что кнопка «🕘 Время» в `/menu`). |

Все ответы бота локализованы (`locales/ru.json`, `locales/en.json`) и идут `reply` на исходное
сообщение с командой. Голая команда без аргументов (`/add`, `/time`, `/lang`, `/remove`) всегда
открывает диалог, а не показывает usage-подсказку — та подсказка остаётся только для непустых
невалидных аргументов (например `/time 9:99` или `/add фигня`).

### Автоматические сообщения

| Событие | Когда приходит |
|---|---|
| Дайджест по карточке | Раз в день, в момент `digest_time` чата (по его таймзоне), если по карточке были изменения (статус/дедлайн/чек-лист), новые комментарии в чате задачи или новые файлы. Одно сообщение на карточку. |
| Недельный пинг | Если у чата есть хотя бы одна активная карточка, но ≥ `WEEKLY_PING_DAYS` (по умолчанию 7) дней не было ни одного содержательного дайджеста и ни одного предыдущего пинга — присылается одна строка вида «по стройкам без изменений с {дата}». |
| Сводка об ошибках | В `ADMIN_CHAT_ID` — только если в прогоне тика были ошибки (сбой Битрикса/LLM/отправки по конкретным карточкам или чатам). |

## 3. Сквозной сценарий

1. Партнёр (или админ) в топике страны пишет `/add 8017`.
2. Бот запрашивает у Битрикса задачу `8017` (`tasks.task.get`), сохраняет alias (заголовок задачи),
   запоминает текущие `last_history_id`/`last_message_id` как стартовый курсор и подтверждает
   подписку: «Отслеживаю: <название> (#8017)».
3. Планировщик тикает каждые `SCHEDULER_TICK_MINUTES` минут (по умолчанию 5) и для каждого
   активного чата проверяет, наступило ли его локальное `digest_time` и не выполнялся ли уже прогон
   сегодня.
4. Перед сбором дельты бот проверяет подзадачи ручных карточек чата (`tasks.task.list` с
   `PARENT_ID`) и сам подписывает топик на новые — они появляются в `/list` с пометкой «↳» и живут
   своей жизнью дальше (отдельные `/remove` тоже возможны).
5. Когда время подошло: бот забирает у Битрикса историю задачи и сообщения чата задачи начиная с
   сохранённого курсора, собирает «дельту» (что изменилось + что написали + какие файлы прислали —
   новая карточка даёт прямую ссылку на файл, старая — ссылку на комментарий с файлом).
6. Если дельта не пустая — дельта уходит в LLM (`prompts/digest.txt`) с просьбой сделать выжимку на
   языке чата; при недоступности LLM после ретраев — fallback на сырой список изменений с пометкой
   «краткая версия без выжимки».
7. Готовый текст (`html.escape`, лимит 4096 символов) отправляется одним сообщением в топик чата;
   только после подтверждённой успешной отправки курсор карточки сдвигается — сбой по одной
   карточке не блокирует и не теряет дельту остальных.
8. Партнёр читает дайджест в своём топике, ничего не открывая в Битриксе.

## 4. Запуск

### Переменные окружения

Скопировать `.env.example` в `.env` и заполнить (см. актуальный список ключей в самом файле):

```bash
cp .env.example .env
```

Ключевые переменные:

- `BITRIX_WEBHOOK_URL` — входящий вебхук Битрикс24 (`https://<portal>/rest/<user_id>/<token>/`).
- `TELEGRAM_BOT_TOKEN` — токен бота от BotFather.
- `OPENAI_API_KEY` / `OPENAI_MODEL` — доступ к LLM для выжимок (модель по умолчанию `gpt-5-mini`).
- `POSTGRES_DSN` — строка подключения к Postgres (в `docker-compose.yml` — `postgres` как хост).
- `SCHEDULER_TICK_MINUTES` — период тика планировщика (по умолчанию 5).
- `DEFAULT_LANGUAGE` — язык нового чата до первого `/lang` (по умолчанию `ru`).
- `ADMIN_CHAT_ID` — telegram id, куда приходят сводки об ошибках прогона (необязательно).
- `WEEKLY_PING_DAYS` — порог тишины для недельного пинга (по умолчанию 7).
- `DRY_RUN` — `true`/`false`. В режиме `DRY_RUN=true` полный пайплайн (Битрикс + LLM) отрабатывает
  как обычно, но вместо отправки в Telegram текст дайджеста печатается в stdout контейнера — удобно
  для обкатки без риска заспамить реальные чаты.

### Локальный запуск

```bash
docker compose up -d --build
docker compose logs -f bot
```

`docker-compose.yml` поднимает два сервиса: `bot` (само приложение, точка входа — `python -m
src.main`) и `postgres` (с healthcheck и лейблом `backup.pgdump=true` для автобэкапа на
MUSPELHEIM). При старте `src/main.py` сам накатывает недостающие миграции из `migrations/` перед
тем как поднять polling и планировщик — отдельного шага миграции руками не требуется.

### Смоук-скрипт Битрикса

`scripts/smoke_bitrix.py` — разовая проверка вебхука по живой карточке (историю задачи, сообщения
чата задачи, файлы, чек-лист) с сохранением обезличенных фикстур в `tests/fixtures/live/`:

```bash
.venv/bin/python -m scripts.smoke_bitrix <task_id>
```

### Тесты

Юнит-тесты, не требующие БД:

```bash
.venv/bin/pytest -v
```

Тесты, помеченные `db` (например `tests/test_db.py`, `tests/test_repo.py`), требуют переменную
`TEST_POSTGRES_DSN`, указывающую на тестовую Postgres (не на продовую БД из `docker-compose.yml`):

```bash
TEST_POSTGRES_DSN=postgresql://bot:bot@<host>:5544/botdb .venv/bin/pytest -v
```

## 5. Эксплуатация

- **Сводки об ошибках.** Если `ADMIN_CHAT_ID` задан, после каждого тика планировщика с ошибками
  (недоступный Битрикс/LLM по конкретной карточке, чат кикнул бота и т.п.) в этот чат/личку
  приходит сообщение со списком проблем за прогон. Пустой прогон без ошибок сводку не шлёт.
- **Ограничение доступа к командам (`restricted`).** Флаг `chats.restricted` по умолчанию `FALSE`
  (команды доступны всем в чате). Если по конкретной стране понадобится ограничить управление
  списком отслеживаемых карточек, флаг переключается и заполняется `chat_admins` — **руками в БД**,
  кода это не касается (сознательное решение MVP — команда управления вайтлистом появится только
  если ограничение реально понадобится):

  ```sql
  UPDATE chats SET restricted = TRUE WHERE id = <chat_id>;
  INSERT INTO chat_admins (chat_id, telegram_user_id) VALUES (<chat_id>, <telegram_user_id>);
  ```

- **Один polling-инстанс.** aiogram long polling не терпит двух одновременно запущенных процессов
  на одном токене — перед стартом на новом окружении (например при переезде с локального
  DRY_RUN-контейнера на MUSPELHEIM) предыдущий инстанс нужно останавливать (`docker compose down`).
- **Ручной сброс дайджеста для теста** — если нужно прогнать дайджест повторно в тот же
  календарный день, не дожидаясь следующих суток: `UPDATE chats SET last_digest_date = NULL WHERE
  id = <chat_id>;`.

### Результаты смоук-теста

Прогнан 2026-07-23 (`scripts/smoke_bitrix.py 42103`, портал `b24.dodoteam.ru` — коробочный):

- Карточки **старого типа**: у задач нет чата → комментарии читаются через `task.commentitem.getlist`
  (курсор `cursors.last_comment_id`, миграция 0002); BB-коды снимаются перед LLM.
- `COMMENT` в `tasks.task.history.list` есть; чек-лист и история читаются штатно.
- `disk.file.get` на файлы из комментариев — ACCESS_DENIED → вместо прямой ссылки на файл
  дайджест даёт ссылку на **комментарий-источник** внутри карточки (`?commentId=<id>#com<id>`,
  `links.comment_url`) — так владелец решил закрыть проблему недоступного `disk.file.get`.
- `tasks.task.list` с `filter: {"PARENT_ID": <id>}` (scope `task`) отдаёт подзадачи родителя —
  проверено живьём: у 42103 подзадача 73689. На этом строится авто-подхват подзадач (§2, `/list`).
- Портал за антиботом Servicepipe: клиент Битрикса ходит **GET-ом с браузерным User-Agent**
  (POST режется JS-challenge'ем). Запрошено WAF-исключение по маркеру `construction-bot` в UA.
- Права вебхука: **«Задачи (task)»** — не путать с соседним «Задачи (tasks)», тот даёт
  `insufficient_scope`. Плюс Диск (disk) и Чат и уведомления (im).
- Живые фикстуры — `tests/fixtures/live/` (только локально, в git не идут — ПДн; редакция
  вычищает URL с токеном вебхука: `DOWNLOAD_URL`/`VIEW_URL`).
