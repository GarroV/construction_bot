# Telegram-бот дайджестов по стройкам из Битрикс24 — Спецификация

> Этот документ самодостаточен. Его можно передать Claude Code (или разработчику) как техническое задание для сборки MVP. Все решения ниже уже согласованы.
>
> **Ревизия 2 (2026-07-21).** Спека вычитана с факт-чеком по официальным докам Bitrix24 и Telegram Bot API + приняты решения владельца. Ключевые изменения против ревизии 1: комментарии читаются из **чата задачи** (`task.commentitem.*` deprecated и не работает в новой карточке), курсор дельты — **per (карточка, чат)** и двигается только после успешной отправки, время дайджеста — **per-country**, хостинг MVP — **сервер MUSPELHEIM** вместо ноутбука, privacy mode бота **остаётся включённым** (Фаза 3 работает на командах и reply, без @упоминаний).

---

## 1. Цель

Бот раз в день собирает апдейты по строящимся пиццериям из карточек-задач Битрикс24 и публикует дайджест в конкретный топик стройки в чате страны (Telegram-супергруппа). Тексты формируются через LLM (OpenAI). Партнёры настраивают бота сами через команды в чате. Поддержка нескольких языков.

## 2. Контекст

- Международная сеть пиццерий, экспансия по странам (сейчас ~24 страны).
- В каждой стране 1–4 строящиеся пиццерии.
- Весь процесс стройки (согласование материалов, обсуждение, файлы) ведётся в **задаче Битрикса**, которую внутри называют «карточкой». Обсуждение идёт в комментариях, к комментариям прикладываются файлы (планировки, интерьерные рендеры и т.п.).
- Также в задаче есть чек-лист — его состояние тоже важно отслеживать.
- Портал — облачный Битрикс, обновляется автоматически: **новая карточка задач (модуль tasks 25.700.0+) уже включена**, комментарии в ней живут в чате задачи, а не в старом comment-API. Это учтено в §7 и §13.

## 3. Модель данных (три уровня)

- **Страна** = супергруппа Telegram + один конкретный топик стройки (`chat_id` + `message_thread_id`).
- **Пиццерия** (например «Бишкек 8», «Бишкек 10») = одна карточка-задача в Битриксе (`bitrix_task_id`).
- Связь: одна страна → много пиццерий. **Один дайджест в день на страну**, внутри — отдельный блок по каждой пиццерии. Всё уходит в один топик стройки.

Пример: чат «Кыргызстан», топик «Стройка» → бот раз в день шлёт дайджест с секциями по Бишкек 8, Бишкек 10, Бишкек 11.

## 4. Функциональные требования

1. Раз в день — в настроенное для страны время (§7, §12) — собирать дельту изменений по каждой отслеживаемой карточке (только новое с прошлого запуска).
2. Отслеживать: новые комментарии (текст + автор + вложения) из чата задачи, изменения чек-листа, смену статуса задачи, дедлайна.
3. Формировать через LLM человеческий дайджест: выжимка обсуждения и выводы (а не дословный пересказ), статус чек-листа, строки про файлы.
4. Публиковать дайджест строго в заданный топик стройки.
5. Партнёры сами добавляют/убирают карточки и меняют язык/время через команды бота.
6. Бот получает из чата **только команды и reply на свои сообщения** (privacy mode Telegram остаётся включённым, §14) — обычная переписка партнёров через бота не проходит вовсе, не логируется и не хранится.
7. **Пустой день — бот молчит.** Если за день ни по одной карточке страны нет изменений, дайджест не отправляется и LLM не вызывается. После 7 дней тишины подряд — один короткий недельный пинг («по стройкам без изменений с {дата}»), чтобы партнёры видели, что бот жив.
8. Мультиязычность (минимум ru/en, в перспективе es, sl и др.).
9. Промпт LLM выносится наружу и меняется без правки кода.
10. (Фаза 3) Отчёт по требованию: команда `/report` — дайджест в моменте, не дожидаясь дневного расписания.

## 5. Команды бота

Команды вводятся прямо в топике стройки. Бот из сообщения с командой автоматически берёт `chat_id` и `message_thread_id` — отдельно ID топика вводить не нужно.

Правила захвата `thread_id`: брать `message_thread_id` **только если** `message.is_topic_message == true`. Иначе (обычная группа, General-топик) сохранять NULL и при отправке параметр опускать. `message_thread_id` из reply-треда — это id треда ответов, не топика; **не сохранять никогда** (иначе отправка упадёт с «message thread not found»).

При первой команде в топике бот сам создаёт запись чата (upsert по `telegram_chat_id + message_thread_id`); поле `country` заполняется названием супергруппы из Telegram (`getChat().title` — де-факто это и есть страна: «Кыргызстан»).

- `/add <task_id>` — добавить карточку к отслеживанию в этом топике. Пример: `/add 8017`.
  - Бот валидирует ID через `tasks.task.get`: задача не найдена / нет доступа → локализованный ответ «задача не найдена или нет доступа»; уже отслеживается → «уже отслеживается».
  - Успех → подтверждение «Отслеживаю: Бишкек 8 (#8017)»; `alias` заполняется **автоматически названием задачи**.
  - Курсор инициализируется **текущими максимальными** `history_id` и `message_id` чата задачи — отслеживание идёт «с этого момента», первый дайджест не вываливает историю стройки за прошлые месяцы. (Стартовое саммари всей истории — отдельная возможная фича, не MVP.) Если у задачи нет чата (старая карточка) — `last_message_id = 0`, комментарии по ней в MVP не собираются (см. fallback в §13).
- `/remove <task_id>` — снять карточку с отслеживания (`active = false`; история связки сохраняется). Повторный `/add` реактивирует карточку и заново инициализирует курсор текущим максимумом. Неотслеживаемый ID → «эта карточка не отслеживается».
- `/list` — показать карточки, отслеживаемые в этом топике.
- `/lang <code>` — язык дайджеста для этого чата (например `/lang ru`, `/lang en`). Неизвестный формат кода → подсказка с примерами.
- `/time <HH:MM> <IANA-таймзона>` — время дневного дайджеста этой страны, например `/time 09:00 Asia/Bishkek`. Оба аргумента валидируются (время по формату HH:MM, таймзона по базе IANA/zoneinfo): невалидный ввод → локализованная подсказка с примером, настройки не меняются. Без второго аргумента меняется только время — но если у чата всё ещё дефолтная `UTC`, бот просит указать таймзону явно (иначе партнёр из Бишкека молча получит «09:00 UTC» = 15:00 местных).
- (Фаза 3) `/report` — дайджест в моменте по этому топику (тот же пайплайн §7, только по запросу; курсоры двигаются так же, но `last_digest_date` **не** трогается — дневной дайджест выходит по расписанию; если `/report` уже съел дельту, дневной может оказаться пустым и не отправиться, §7 п.5). Плюс бот отвечает на reply к своим сообщениям — это работает при включённом privacy mode.

## 6. Права доступа

- По умолчанию команды доступны **всем** участникам чата (партнёрские команды — взрослые люди, справятся).
- В схеме заложен флаг `restricted` на уровне чата. Значение по умолчанию — «открыто». Если по какой-то стране понадобится ограничить, флаг переключается, и команды начинают слушаться только вайтлиста (`chat_admins`). **Код при этом не меняется.**
- Для MVP переключение `restricted` и наполнение `chat_admins` — **руками в БД** (SQL от владельца бота). Это осознанное решение: команды управления вайтлистом появятся, только если ограничение реально понадобится.

## 7. Как работает дневная дельта (ядро Фазы 1)

**Планировщик.** Джоба тикает каждые `SCHEDULER_TICK_MINUTES` (5 минут). На каждом тике по каждому активному чату вычисляется локальное время (`timezone` чата): если локальное время ≥ `digest_time` и `last_digest_date IS NULL OR last_digest_date < текущая локальная дата`, запускается сбор по карточкам чата. (Предикат именно `<`: при смене таймзоны через `/time` с откатом локальной даты возможен пропуск одного дня — принято осознанно, дельта не теряется и уедет со следующим дайджестом.) Прогоны не накладываются: джоба в один поток (APScheduler `max_instances=1`, §11). Побочный бонус: это же даёт catch-up — если сервер был выключен в назначенный час, дайджест уйдёт на первом тике после подъёма.

**По каждой карточке чата:**

1. **История:** `tasks.task.history.list`. Внимание: фильтра «новее ID» у метода **нет** (filter умеет только тип события) — серверного среза «после курсора» не существует. Дельта режется на клиенте: читаем страницами (50/страницу, порядок по умолчанию desc), останавливаемся, как только id записи ≤ `last_history_id`. Из истории берём: изменения чек-листа (`CHECKLIST_ITEM_CREATE/RENAME/REMOVE/CHECK/UNCHECK`), смену статуса, дедлайна и т.п.
2. **Комментарии — из чата задачи** (основной путь для новой карточки): `tasks.task.get` c `select: ["chat.id"]` → `im.dialog.messages.get` (`DIALOG_ID = "chatN"`, `FIRST_ID = last_message_id` — курсор встроен в метод, `LIMIT ≤ 50`; вложения приходят в блоке `files`: id, name, size). Системные сообщения чата (смена статуса/ответственного и т.п.; признак system/`author_id = 0`) отфильтровываются — иначе они задублируют события истории из п.1. `task.commentitem.get/getlist` — **deprecated, в новой карточке не работают**; fallback для старых карточек — вне MVP, решение после смоук-теста (§13).
3. **Чек-лист, текущее состояние:** `task.checklistitem.getlist` (посчитать выполнено/всего по `IS_COMPLETE`).
4. **Файлы:** id вложения → `disk.file.get` → постоянный `DETAIL_URL` (подробности и запреты — §8).
5. **Пустая дельта:** если по всем карточкам чата ничего нового — LLM не вызывается, ничего не отправляется, `last_digest_date` помечается (день обработан). Если по части карточек пусто, а по части нет — дайджест уходит; пустые пиццерии получают одну строку «изменений нет» из locale-файла (тоже без вызова LLM).
6. **LLM:** сырьё по пиццерии (обсуждение + изменения) → OpenAI. Ретраи: 2–3 попытки с экспоненциальным backoff. Финальный отказ → **fallback**: отправить сырой структурированный список изменений (авторы + первые строки комментариев, чек-лист N/M, файлы со ссылками) с пометкой «краткая версия без выжимки» + алерт в `ADMIN_CHAT_ID`. Дайджест деградирует, но не исчезает молча.
7. **Отправка:** **одно сообщение на пиццерию** (лимит 4096 символов, §14), последовательно в топик.
8. **Курсоры:** у каждой связки (карточка, чат) свой курсор (`last_history_id`, `last_message_id`). Курсор двигается **только после подтверждённой успешной отправки** сообщения этой карточки в этот чат — семантика at-least-once: редкий дубль при падении между отправкой и записью курсора приемлем, молчаливая потеря дельты — нет. Сбой по одной карточке/чату не блокирует и не теряет дельту остальных. Кэшировать в рамках прогона можно **только сырые страницы** истории/сообщений по task_id (читая от минимального курсора среди чатов этой карточки); резать дельту — по курсору каждого чата отдельно: у чатов разные стартовые точки, и чужой срез молча теряет или дублирует события.
9. **Завершение прогона по чату:** независимо от частичных сбоев по отдельным карточкам выставляется `last_digest_date` (текущая локальная дата), а если отправлено хоть одно содержательное сообщение (включая fallback) — обновляется `last_posted_at`. Упавшие карточки до завтра не ретраятся: их курсоры не сдвинуты, дельта не потеряна и уедет со следующим дайджестом, а сбой попадает в сводку `ADMIN_CHAT_ID`. Это исключает 5-минутный ретрай-спам по чату со стабильной ошибкой.

**Недельный пинг:** если у чата есть активные карточки и ≥ `WEEKLY_PING_DAYS` (7) дней не было ни содержательного сообщения (`last_posted_at`), ни предыдущего пинга (`last_ping_at`; для нового чата опора — `created_at`), то на тике его `digest_time` бот шлёт одну строку «по стройкам без изменений с {дата}», где `{дата}` = `last_posted_at` (начало тишины; если NULL — дата подключения чата). Пинг обновляет `last_ping_at`, но **не** `last_posted_at` — дата начала тишины не дрейфует от пинга к пингу.

> Шаг 6 — это тот «слой над conversations»: LLM читает переписку и делает выводы, а не пересказывает дословно.

## 8. Файлы

Идём простым путём — **не читаем содержимое**, а сигнализируем и даём ссылку:

- В дайджесте, там где упоминается **пиццерия** (заголовок секции) — ссылка на **карточку задачи**. Ни один REST-метод URL карточки не возвращает — бот конструирует его сам: хост портала (из `BITRIX_WEBHOOK_URL`) + `/company/personal/user/<user_id владельца вебхука>/tasks/task/view/<task_id>/`. Кликабельность проверяется на смоук-тесте Фазы 0 (§13).
- Там где упоминается **файл** — ссылка на **сам файл** (`DETAIL_URL`).

Важно про ссылки Битрикса:

- **`DETAIL_URL` возвращает только `disk.file.get`** — постоянная страница файла на портале, открывается после входа в Битрикс (у партнёров доступ есть). **Использовать её.** `disk.attachedObject.get` поле `DETAIL_URL` не отдаёт — он нужен только для маппинга вложения на файл (`OBJECT_ID`). Рабочая цепочка: `FILE_ID`/`OBJECT_ID` вложения → `disk.file.get` → `DETAIL_URL`.
- `DOWNLOAD_URL` из вложений партнёрам **не давать — вдвойне**: ссылка не только протухает, но при вызовах через входящий вебхук содержит **незатухающий токен вебхука** — опубликовать её значит слить секрет с полным доступом к API. Это касается и `urlDownload` из блока `files` чата задачи.
- Для файлов из чата задачи проверить на смоук-тесте Фазы 0, что их id резолвится через `disk.file.get` (см. §13).

Чтение содержимого PDF/картинок через OpenAI — опция на будущее (Фаза 4). Архитектурно оставить под неё место, но в MVP не включать. CAD-форматы (DWG/DXF) OpenAI напрямую не читает — если понадобится, потребуется конвертация.

## 9. Промпт LLM

Хранится **вне кода — файлом в репозитории** (`prompts/digest.txt`): версионирование даёт git, бот перечитывает файл при старте процесса. Меняешь текст промпта — код не трогаешь. (Таблица `prompts` в Postgres из MVP убрана — вернётся на Фазе 4, если понадобится править промпт без доступа к серверу; два канонических места хранения = гарантированный дрифт.)

Черновик (подгонится под идеальный формат позже):

```
Ты формируешь дневной апдейт по стройке пиццерии для партнёрского чата.
Пиши на языке: {language}. Тон — деловой, короткий, по делу.
Пиши компактно: не более ~150 слов на весь апдейт.

Дата: {date}
Пиццерия: {pizzeria_name}
Статус чек-листа: {checklist_done}/{checklist_total} выполнено
Изменения задачи (статус, дедлайн, чек-лист): {task_changes}

Обсуждение за день (автор -> сообщение):
{comments}

Приложенные файлы: {files}

Сделай:
1. 2-4 предложения: что обсуждали и к чему пришли (выводы, а не пересказ).
2. Отдельной строкой — заметные решения/блокеры, если есть.
3. Список приложенных файлов с названиями (ссылки подставит система).
Не выдумывай фактов.
```

Плейсхолдеры `{...}` подставляет бот. Вызов LLM происходит **только при непустой дельте** (§7 п.5), поэтому случая «за день ничего не было» в промпте нет. Вывод LLM ограничивается `max_tokens` и прогоняется через `html.escape`; ссылки на карточку и файлы система подставляет после генерации, оборачивая в `<a href="DETAIL_URL">имя</a>` (LLM их не выдумывает).

## 10. Языки — два слоя, одна настройка

- **Язык дайджеста** (что бот пишет в топик): поле `digest_language` чата, подставляется в `{language}` промпта. Новый язык не требует ничего — LLM просто пишет на нём. Управляется командой `/lang`.
- **Язык интерфейса бота** (ответы на команды): locale-файлы (`ru.json`, `en.json`, …). Новый язык = один файл перевода.
- Для MVP оба слоя управляются **одним полем** `digest_language`: интерфейс использует locale-файл этого языка, а если файла нет (например `sl` для дайджеста есть, `sl.json` ещё не написан) — fallback на `en`. Отдельная колонка `interface_language` появится, только если слои реально разойдутся.

Пример: Кыргызстан получает апдейты на русском, Словения — на словенском (ответы на команды — на английском, пока нет `sl.json`), код один.

## 11. Архитектура и эксплуатация

- **Хостинг MVP — домашний сервер MUSPELHEIM** (Windows + Docker Desktop, постоянно включён): машина как раз настроена под TG-ботов на long polling — Docker, Postgres и ночные бэкапы уже есть. Проект — `C:\projects\construction_bot` копией из `C:\projects\_template`. Контейнер Postgres пометить `backup.pgdump=true` — тогда он попадает в штатный ночной автобэкап (03:30, ретеншн 14 дней). После проверки MVP — переезд на серверы компании (Фаза 4).
- **Сервис на Python**: `aiogram` 3.x (Telegram, long polling) + `httpx` (Битрикс) + `openai`.
- **Postgres 15+** — конфиг связок и курсоры дельты (в §12 используется `UNIQUE NULLS NOT DISTINCT`).
- **Планировщик** — APScheduler `AsyncIOScheduler` в том же event loop, что и polling (один процесс, без cron). Тик каждые 5 минут, per-chat время — §7; `coalesce=true` от накопления пропущенных тиков, `max_instances=1` — прогоны не накладываются (при 2 rps Битрикса полный прогон может длиться дольше тика).
- **Один polling-инстанс на токен**: два одновременно работающих процесса дадут конфликт `getUpdates`, и команды начнут теряться. При переезде на серверы компании — сначала остановить старый процесс, потом запускать новый.
- **Обработка ошибок:**
  - *Telegram:* `429 Too Many Requests` → экспоненциальный backoff по `retry_after` (aiogram кидает `TelegramRetryAfter`); `migrate_to_chat_id` → молчаливая смена chat_id при апгрейде группы до супергруппы, ловить (`TelegramMigrateToChat`) и обновлять сохранённый ID; «message is too long» при пер-пиццерийной отправке возникать не должен, страховка — жёсткий сплит по границам строк/тегов; бот выкинут из чата — событие `my_chat_member` (kicked/left) или `TelegramForbiddenError` при отправке → `chats.active = false` + алерт в `ADMIN_CHAT_ID` (планировщик не долбится в мёртвый чат до ручной реактивации).
  - *Битрикс:* троттлинг клиента ≤ 2 rps (leaky bucket облака, порог блокировки 50); HTTP `503 QUERY_LIMIT_EXCEEDED` → retry с backoff; `batch` до 50 команд = 1 обращение к лимиту — использовать для опроса 50–80 задач.
  - *OpenAI:* ретраи + fallback на сырой список изменений (§7 п.6).
- **Наблюдаемость:** `ADMIN_CHAT_ID` (личка владельца или служебный чат) — после каждого прогона с ошибками бот шлёт туда сводку («отправлено 21/24; ошибки: Словения — chat not found, …»). Структурные логи по чату: сколько карточек, сколько событий, статус отправки.
- **Dry-run:** `DRY_RUN=true` — полный пайплайн (Битрикс + LLM), но вывод в stdout вместо Telegram. Для обкатки без спама в чаты.

## 12. Схема БД (Postgres 15+, черновик DDL)

```sql
CREATE TABLE chats (
    id                 BIGSERIAL PRIMARY KEY,
    country            TEXT,                          -- заполняется названием супергруппы при первом контакте
    telegram_chat_id   BIGINT NOT NULL,
    message_thread_id  BIGINT,                        -- NULL = чат без топиков / General
    digest_language    TEXT NOT NULL,                 -- при создании чата берётся из DEFAULT_LANGUAGE (.env)
    digest_time        TIME NOT NULL DEFAULT '09:00', -- локальное время страны
    timezone           TEXT NOT NULL DEFAULT 'UTC',   -- IANA, напр. 'Asia/Bishkek'
    last_digest_date   DATE,                          -- дедуп дневного прогона (локальная дата чата)
    last_posted_at     TIMESTAMPTZ,                   -- последний содержательный дайджест (пинг его НЕ двигает)
    last_ping_at       TIMESTAMPTZ,                   -- последний недельный пинг
    restricted         BOOLEAN NOT NULL DEFAULT FALSE,
    active             BOOLEAN NOT NULL DEFAULT TRUE, -- false = бот удалён из чата / чат выключен вручную
    created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE NULLS NOT DISTINCT (telegram_chat_id, message_thread_id)
);

CREATE TABLE cards (
    id              BIGSERIAL PRIMARY KEY,
    bitrix_task_id  BIGINT NOT NULL,
    chat_id         BIGINT NOT NULL REFERENCES chats(id) ON DELETE CASCADE,
    alias           TEXT,                          -- автоматически = название задачи при /add
    added_by        BIGINT,                        -- telegram user id
    active          BOOLEAN NOT NULL DEFAULT TRUE, -- /remove -> false; повторный /add реактивирует
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (bitrix_task_id, chat_id)
);

CREATE TABLE cursors (
    bitrix_task_id  BIGINT NOT NULL,
    chat_id         BIGINT NOT NULL,
    last_history_id BIGINT NOT NULL,               -- init текущим max при /add
    last_message_id BIGINT NOT NULL,               -- курсор чата задачи (FIRST_ID); 0 = у задачи нет чата (§5, §13)
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (bitrix_task_id, chat_id),
    FOREIGN KEY (bitrix_task_id, chat_id)
        REFERENCES cards (bitrix_task_id, chat_id) ON DELETE CASCADE
);

CREATE TABLE chat_admins (
    chat_id          BIGINT NOT NULL REFERENCES chats(id) ON DELETE CASCADE,
    telegram_user_id BIGINT NOT NULL,
    PRIMARY KEY (chat_id, telegram_user_id)
);
```

Почему так:

- `UNIQUE NULLS NOT DISTINCT` — иначе два ряда одного чата с `message_thread_id = NULL` оба вставятся (в Postgres NULL ≠ NULL) и страна получит двойные дайджесты.
- Курсор — **per (карточка, чат)**, а не глобальный на задачу: если одну карточку отслеживают два чата (типовой случай — тестовый чат + боевой), глобальный курсор означает, что первый обработанный чат «съедает» дельту второго. Составной FK на `cards` каскадно чистит курсор при удалении связки.
- Таблицы `prompts` в MVP нет — промпт живёт файлом (§9).

## 13. Битрикс24 — API

Доступ через **входящий вебхук** (URL вида `https://<portal>/rest/<user_id>/<token>/<method>`). Права: **задачи + диск + чаты (im)** — без `im` не прочитать комментарии новой карточки. Вызовы идут от имени пользователя-владельца вебхука: он должен видеть все отслеживаемые задачи (участник/наблюдатель/руководитель).

Нужные методы:

- `tasks.task.get` — карточка, статус, ответственный, дедлайн; через `select: ["chat.id"]` — id чата задачи; валидация `/add` (§5).
- `tasks.task.history.list` — история изменений задачи (создание, `CHECKLIST_ITEM_*`, смена полей) с датой и автором. Фильтр — **только по типу события**, среза «новее ID» нет: дельта режется на клиенте (§7 п.1).
- `im.dialog.messages.get` — сообщения чата задачи (`DIALOG_ID = "chatN"`; `FIRST_ID` — курсор «сообщения новее ID», `LIMIT ≤ 50`; вложения в блоке `files`). **Основной источник комментариев** на новой карточке. Требует scope `im`.
- `task.commentitem.get` / `task.commentitem.getlist` — **deprecated с tasks 25.700.0, в новой карточке не работают.** Fallback для старых карточек — **вне схемы MVP**: решение после смоук-теста; если живые старые карточки реально встретятся, в `cursors` добавится nullable `last_comment_id` (id комментариев — другое пространство идентификаторов, чем сообщения чата).
- `task.checklistitem.getlist` — пункты чек-листа с состоянием `IS_COMPLETE`.
- `disk.attachedObject.get` — маппинг вложения на файл (`OBJECT_ID`); `disk.file.get` — параметры файла, в т.ч. постоянный `DETAIL_URL` (только этот метод его отдаёт).

**Смоук-тест на Фазе 0 (обязателен, на живой карточке портала):** (а) читаются ли сообщения чата задачи через `im.dialog.messages.get`; (б) отражаются ли они в `tasks.task.history.list` как `COMMENT` (доки молчат — если да, историю можно использовать как единый триггер); (в) резолвятся ли файлы из чата задачи через `disk.file.get`; (г) кликабелен ли сконструированный URL карточки (§8) под партнёрским аккаунтом; (д) как выглядят системные сообщения в выдаче чата и по какому признаку их отличать (§7 п.2). Заодно снять JSON-фикстуры реальных ответов для тестов (§16 п.6).

Лимиты REST (облако): leaky bucket **2 запроса/сек**, порог блокировки 50; при превышении — HTTP `503 QUERY_LIMIT_EXCEEDED`; пагинация `*.list` — 50 записей на страницу (`start`); `batch` — до 50 команд за вызов, считается одним обращением.

## 14. Telegram — API

- Отправка в конкретный топик: параметр `message_thread_id` в `sendMessage`. Без него сообщение уходит в General. Правила захвата thread_id из входящих — §5.
- **Privacy mode остаётся включённым** (дефолт BotFather): бот получает только команды, reply на свои сообщения и сервисные события — переписка партнёров до бота физически не доходит. Этого достаточно для всех фаз: команды (Фазы 0–2), `/report` и ответы на reply (Фаза 3). @упоминания бота осознанно не поддерживаются (потребовали бы отключить privacy mode и пере-добавить бота во все чаты).
- Бот должен быть добавлен в супергруппу; для отправки в известный топик достаточно членства, админка не нужна.
- **Лимит длины: 4096 символов** на сообщение («after entities parsing»). Поэтому дайджест отправляется **по одному сообщению на пиццерию**; при превышении одним блоком — сплит по границам строк, не разрывая HTML-теги. 24 страны × 3–4 сообщения укладываются в лимиты (~20 сообщений/мин на группу) с большим запасом.
- **Форматирование: `parse_mode=HTML`.** Текст LLM экранируется `html.escape`, ссылки — `<a href="...">`. MarkdownV2 не использовать: он требует экранирования 18 спецсимволов, и LLM-текст с точками/скобками/дефисами гарантированно даёт `400 can't parse entities`.
- Лимиты: ~20 сообщений/мин на одну группу, 30/сек суммарно. При превышении — `429` с `retry_after`, экспоненциальный backoff.
- `migrate_to_chat_id`: при апгрейде группы до супергруппы `chat_id` меняется молча — ловить и обновлять в БД.

## 15. Фазы реализации

- **Фаза 0 — каркас:** входящий вебхук Битрикса (права: задачи + диск + **im**), **смоук-тест API на живой карточке** (§13) + снятие JSON-фикстур, токен бота у BotFather (privacy mode не трогать — остаётся включённым), docker-compose на MUSPELHEIM, схема БД.
- **Фаза 1 — ядро дайджеста:** планировщик с per-chat временем, дельта (история + чат задачи, клиентская резка по курсору), LLM с файлом-промптом + ретраи + fallback, отправка по-пиццерийно (HTML), сдвиг курсоров после отправки, молчание в пустой день + недельный пинг, `ADMIN_CHAT_ID`-отчёты, `DRY_RUN`. Сидинг данных для проверки — базовым `/add` (собирается вместе с ядром, §16 п.4). **Это работающее ядро; полный MVP = Фазы 0–2 (§16).**
- **Фаза 2 — самообслуживание:** доводка команд `/add /remove /list /lang /time` до партнёрского качества — валидация, локализованные ответы, авто-создание чата при первой команде; флаг `restricted` + вайтлист (управление — руками в БД, §6).
- **Фаза 3 — отчёт по требованию:** `/report` (дайджест в моменте) + ответы на reply к сообщениям бота.
- **Фаза 4 — доводка:** таблица `prompts` и редактирование промпта без доступа к серверу, новые языки, опционально чтение PDF, переезд на серверы компании (сначала стоп старого polling-инстанса, потом старт нового).

## 16. Порядок сборки MVP — Фазы 0–2 (для Claude Code)

1. `docker-compose` с сервисом бота + Postgres 15+ (метка `backup.pgdump=true` для автобэкапа MUSPELHEIM); `.env` под токены.
2. Схема БД по разделу 12 + миграции.
3. Клиент Битрикса: `tasks.task.get` (+`chat.id`), `history.list` (клиентская резка дельты), `im.dialog.messages.get`, `checklistitem.getlist`, `disk.attachedObject.get`, `disk.file.get`; троттлинг 2 rps, retry на 503, `batch`. Смоук-скрипт по живой карточке + снятие фикстур (§13).
4. Telegram-бот на aiogram: long polling, команды `/add /remove /list /lang /time`, захват chat_id + thread_id по правилам §5, upsert чата, локализованные ответы из locale-файлов (§10), проверка `restricted`/вайтлиста (§6).
5. Джоба дайджеста: тик 5 мин → per-chat время → дельта → LLM (файл-промпт, `html.escape`, `max_tokens`) → отправка по-пиццерийно → сдвиг курсора после отправки; fallback OpenAI, отчёты в `ADMIN_CHAT_ID`, недельный пинг.
6. Тесты: юнит-тесты парсера истории/дельты на JSON-фикстурах с живой карточки; `DRY_RUN`-прогон полного пайплайна; тестовый чат-«страна» с тестовым топиком — всё обкатывается там до подключения первого партнёрского чата.
